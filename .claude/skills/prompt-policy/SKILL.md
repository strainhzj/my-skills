# Prompt Policy

A lightweight skill for governing the constraint layer of a generated prompt.
Reduces prompt drift, duplicated rules, and conflicting guardrails across an agent loop.

## Core Principle

**Task intent is immutable; constraints are mergeable.**

- Do not rewrite the user's goal, router's task intent, or requested deliverable
- Only inspect, normalize, merge, prioritize, or append constraints
- Do not add new business requirements or override explicit user intent
- Do not expand the prompt with generic filler rules ("be careful", "do your best")

## Constraint Handling Model

This skill operates on two layers:

### Layer 1: Global Constraints

Rules that apply broadly across task types, such as:

- do not fabricate facts
- mark uncertainty explicitly
- preserve user intent
- avoid destructive actions without confirmation

### Layer 2: Task-Specific Constraints

Rules that depend on the task type, such as:

- code_review: separate confirmed defects from hypotheses
- research: distinguish sourced claims from inference
- service_management: avoid restart/delete/stop actions without confirmation
- deployment: report assumptions and deployment scope clearly

## Fragile Operation Rule

Constraint definitions and constraint mappings are treated as **fragile policy data**.
Do not improvise, regenerate, or mutate them inline unless the user explicitly asks to update the policy set.

## Workflow

### Step 1: Preserve task intent

Identify the task body and freeze it (see Core Principle).

### Step 2: Inspect existing constraint layer

Look for sections or phrases such as:

- Constraints, Rules, Guardrails, Requirements
- Notes, Safety, Review Criteria, Output Rules

If none exist, continue with an empty constraint set.

### Step 3: Load static policy data

Read the JSON rule files and select:

- global constraints
- task-specific constraints for the detected task type
- optional priority metadata if defined

Do not fabricate new policy entries when the JSON files do not contain them.

### Step 4: Normalize and merge

1. Compare existing constraints against selected policy rules
2. Remove exact duplicates
3. Merge near-duplicates when the meaning is the same
4. Keep the stricter rule when two rules are compatible but unequal
5. If two rules conflict, preserve the one with higher priority from the static policy files
6. If user intent explicitly requires an exception, do not silently override it; surface the tension clearly

### Step 5: Decide whether to inject

- If sufficient equivalent constraints already exist, do not append more
- If critical constraints are missing, append only the missing items
- If no critical constraints are missing and no conflict or duplication is detected, bypass this skill

### Step 6: Return compact output

Return the original task body unchanged plus one normalized constraint block when needed.
Keep the constraint section compact and priority-ordered.

## Output Format

Use this structure when constraints are needed:

```markdown
[Task]
<original task body, unchanged>

[Constraints]
Priority 1 - Critical
- ...

Priority 2 - Task-Specific
- ...

Priority 3 - Output / Style
- ...
```

If no injection is needed, return the original prompt unchanged.

## Task Type Guidance

### code_development
Focus on preserving task scope, avoiding unsafe assumptions, and requiring explicit mention of risky changes when relevant.

### code_review
Focus on separating confirmed issues from suspicions, avoiding fabricated defects, and keeping findings evidence-based.

### research
Focus on uncertainty labeling, source-grounded claims, and separation of fact from synthesis.

### service_management
Focus on confirmation before destructive operations and clear reporting of operational risk.

### deployment
Focus on scope clarity, environment assumptions, and avoiding silent production-impacting actions.

### document_generation
Focus on fidelity to source intent and avoiding invented details.

## Decision Standard

This skill should behave like a **policy governor**, not a creative optimizer.
If uncertain, prefer:

- minimizing added text
- skipping injection rather than adding weak generic rules
