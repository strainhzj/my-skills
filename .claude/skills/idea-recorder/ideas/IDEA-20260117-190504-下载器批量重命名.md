# 下载器批量重命名功能

## 📝 用户原想法
> 我希望能为下载器添加批量重命名的功能

## 🧠 项目认知分析

### 技术关联
- **涉及模块**:
  - 后端: `app/api/endpoints/torrents.py` (种子管理 API)
  - 前端: `btpManager_fronted/src/views/torrents/` (种子管理页面)
  - 数据模型: `app/models/torrent.py` (种子数据模型)

- **技术栈影响**:
  - 后端: FastAPI 路由扩展,需要新增批量操作 API
  - 前端: Vue 2 + Element UI 表格批量选择功能
  - 通信: WebSocket 实时状态同步需要支持批量更新

- **架构影响**:
  - API 层: 新增 `POST /api/torrents/batch-rename` 端点
  - 服务层: 新增批量重命名业务逻辑,处理事务和异常
  - 前端组件: 扩展种子列表组件,添加批量选择和操作 UI

### 实现难度
- **前端工作量**: 中等 (2-3天)
  - Element UI 表格批量选择功能 (已有基础)
  - 批量操作对话框和表单设计
  - 进度提示和错误处理

- **后端工作量**: 中等 (2-3天)
  - 批量重命名 API 端点实现
  - 事务处理和错误回滚
  - 与下载器 API 的交互 (qBittorrent/Transmission)

- **数据库变更**: 无需变更
  - 种子信息主要存储在下载器中
  - 本地数据库仅缓存状态,无需额外字段

- **风险等级**: 🟡 中等风险
  - 重命名冲突处理 (同名文件处理策略)
  - 批量操作的事务一致性
  - 下载器 API 的兼容性 (不同版本支持程度)

## ❓ 疑问点

1. **重命名规则**: 是提供规则模板 (如 `{序列号}-{原文件名}`) 还是完全自定义?
2. **冲突处理**: 如果目标文件名已存在,是否自动重命名或跳过?
3. **执行策略**: 是同步执行 (可能超时) 还是异步任务 (后台执行,进度通知)?
4. **范围限制**: 是否需要限制单次批量操作的数量 (防止性能问题)?
5. **权限控制**: 是否需要特殊权限才能执行批量重命名?
6. **操作日志**: 是否需要记录批量操作的历史,支持撤销?

## 💡 改进建议

### 功能设计
1. **支持多种重命名模式**:
   - 前缀/后缀添加
   - 序列号自动生成 (001, 002, ...)
   - 正则表达式替换
   - 完全自定义

2. **预览机制**:
   - 实施前显示变更预览
   - 高亮显示冲突和问题
   - 允许用户调整后再执行

3. **分批执行**:
   - 大批量操作自动分批 (如每批 50 个)
   - 显示实时进度 (如 "已完成 45/100")
   - 支持暂停和继续

4. **错误处理**:
   - 部分失败不影响其他文件
   - 详细的错误日志
   - 支持重试失败项

### 技术实现
1. **后端实现要点**:
   ```python
   # 批量重命名 API 示例
   @router.post("/torrents/batch-rename")
   async def batch_rename_torrents(
       request: BatchRenameRequest,
       current_user: User = Depends(get_current_user)
   ) -> CommonResponse:
       """
       批量重命名种子文件

       - 验证用户权限
       - 执行重命名预览
       - 分批处理 (每批 50 个)
       - 返回执行结果和错误日志
       """
   ```

2. **前端实现要点**:
   - 使用 Element UI 的 `<el-table>` 多选功能
   - 批量操作按钮固定在页面顶部或底部
   - 进度对话框使用 `<el-progress>` 显示实时进度

3. **WebSocket 通知**:
   - 使用现有的 WebSocket 推送机制
   - 推送类型: `batch_rename_progress`
   - 数据格式: `{ total, completed, failed, current }`

### 性能优化
1. **异步执行**:
   - 使用 APScheduler 创建后台任务
   - 任务 ID 返回给前端,支持查询进度

2. **缓存机制**:
   - 预览结果缓存 5 分钟
   - 避免重复计算

3. **并发控制**:
   - 限制同时执行的批量操作数量
   - 队列机制处理多个请求
